# Robot Fault Diagnosis SOP（tech-internal）

## 1. 适用范围
本 SOP 仅适用于技术部内部（`tech-internal`）的机器人故障诊断流程。用于回答“技术部机器人故障诊断流程是什么”。

## 2. 目标
- 统一技术部故障诊断步骤。
- 提升排障效率与记录一致性。
- 确保跨团队交接时信息完整可追溯。

## 3. 标准流程
1. 故障接收  
记录故障来源、机器人 ID、现场影响范围、发生时间和现象描述，生成技术诊断任务。

2. 日志与遥测检查  
检查控制面日志、任务执行日志、遥测指标（电量、温度、网络时延、关键告警）并标注异常时间点。

3. 网络连接检查  
确认机器人与网关/API 通信状态，核对 DNS、端口连通性、丢包率与心跳稳定性。

4. 传感器/电机/导航模块排查  
按模块执行自检：传感器健康、驱动电机状态、导航定位与路径规划约束，定位是硬件、固件还是配置问题。

5. SDK/API 调用检查  
核对 SDK 版本、API 请求参数、权限范围、错误码、超时、重试策略和任务状态，确认是否由调用方式或接口配置导致故障。

6. 复现与回归测试  
在测试环境复现故障，记录可重复步骤；修复后执行导航、传感器、电机、任务调度和日志上报的回归测试。

7. 远程重启或现场派单  
可远程恢复时先执行受控重启与参数回滚；不能远程恢复时，创建现场派单并标注优先级与安全注意事项。

8. 记录到技术工单  
将诊断结论、临时处置、根因分类、后续动作和责任人完整记录到技术工单，便于复盘与知识沉淀。

## 4. 升级规则
- P1 故障：立即升级值班负责人并进入持续跟踪。
- P2 故障：当日闭环并同步影响评估。
- P3 故障：纳入常规排期处理并补充观察数据。
- 连续两轮排查仍无法定位，或涉及固件、导航算法、核心安全策略时，升级给高级工程师。

## 5. 权限边界
- 本文不包含销售报价、客户投诉明细、人事信息。
- 本文仅描述技术诊断流程，不对外公开。

## 6. 可用于问答的明确事实点
- 技术部机器人故障诊断流程包括：故障接收、日志与遥测检查、网络连接检查、传感器/电机/导航模块排查、SDK/API 调用检查、复现与回归测试、远程重启或现场派单、记录到技术工单。
- 该流程属于 `tech-internal`，不是 `public-policy` 或 `company-internal`。
- “传感器、电机、导航模块异常怎么检查”应回答模块自检、定位约束、硬件/固件/配置判断和回归测试。
