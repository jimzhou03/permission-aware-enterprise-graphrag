# 产品需求评审流程（product-internal）

## 1. 适用范围
本文件属于星海智造机器人有限公司 Demo v0.9.4 的虚构产品部内部资料，只允许产品部和管理员访问。本文用于回答“产品需求如何评审”“版本发布流程是什么”等问题。

## 2. 需求收集
产品部从销售、客服、市场、技术、交付和客户反馈中收集需求。每条需求需要记录来源、目标用户、业务场景、预期价值、紧急程度和权限影响。未经确认的客户口头要求不能直接进入版本承诺。

## 3. PRD 与评审
产品经理编写 PRD，说明目标、范围、用户流程、异常场景、验收标准、数据与权限边界。评审流程包括产品自审、原型评审、技术评审、测试评审和业务确认。高风险需求需要拆分灰度方案和回滚条件。

## 4. 版本发布
版本发布流程包括需求冻结、开发联调、测试回归、灰度发布、指标观察、正式发布和复盘。发布说明只能使用已确认能力，不得写入内部路线图或竞品敏感结论。问题复盘进入下一轮需求池。

## 5. 可回答的问题
- 产品需求如何评审？
- 版本发布流程是什么？
- PRD 应包含哪些内容？
- 原型评审和技术评审看什么？
- 未确认需求能不能对外承诺？
